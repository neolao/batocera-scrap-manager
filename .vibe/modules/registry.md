# Module: registry
**Role:** Centralized index of already-known games (scraped metadata and media), populated and kept up to date by importing the ROMs folders' `gamelist.xml` files without duplicating known entries, while detecting metadata that changed and copying the referenced media.
**Files:** `internal/registry/registry.go`
**Exports:**
- `registry.Entry{System, Game}`
- `registry.Registry{Entries}`
- `registry.Load(path string) (*Registry, error)` — path is the registry folder; reconstructs the registry by scanning its per-system subfolders for game JSON files; missing folder = empty registry, no error
- `registry.Save(path string, reg *Registry) error` — writes one JSON file per game (named after the ROM's base name, extension stripped) inside its system's subfolder, creating folders as needed
- `(*Registry).Import(system string, games []gamelist.Game) (added, updated, unchanged int)` — deduplicated by (system, gameID) key, via `indexOf` (matches on `gameID(Game.Path)` — base name, directory prefix *and file extension* both stripped — see [`decisions/005`](../decisions/005-match-registry-entries-by-rom-filename-not-full-path.md) and [`decisions/014`](../decisions/014-dedupe-by-extension-stripped-filename-too.md)); if the entry already exists, compares the full metadata and replaces + counts "updated" on a difference, otherwise "unchanged"
- `registry.ImportFromRomsFolder(reg *Registry, romsFolder, registryFolder string, onProgress func(ProgressEvent)) (added, updated, unchanged int, err error)` — scans the subfolders (one per Batocera system) of a ROMs folder, reads `gamelist.xml` if present (otherwise silently skips the system), imports each game, and for every added or updated game copies its referenced media (cover art, video, marquee, thumbnail) into `registryFolder`, mirroring the Batocera per-system arborescence; unchanged games are not recopied. A game with neither a description nor an image (`hasScrapedData`) is skipped entirely — not imported, not counted, no progress event — since it carries no scraped information worth keeping (see [`decisions/007`](../decisions/007-skip-empty-games-only-on-import-not-retroactively.md)); this only affects new imports, an already-known entry is never removed retroactively if its local sheet later loses its last data. If non-nil, `onProgress` is called once per game actually imported (added, updated, or unchanged) with its system, 1-based index and total count within that system's full local game list (including skipped games), and name
- `registry.ProgressEvent{System, GameIndex, GameCount, GameName}` — one progress notification emitted by `ImportFromRomsFolder` and `ImportGame`. `CompletionEvent` (below) is a type alias for `ProgressEvent`, not a separate struct — both flows describe the same shape (a game's position within its system's local game list)
- `registry.ImportGame(reg *Registry, romsFolder, registryFolder, system, romFilename string, onProgress func(ProgressEvent)) (added, updated, unchanged int, err error)` — the single-game counterpart of `ImportFromRomsFolder`: imports the one local game entry matching (system, romFilename) — matched by ROM base name via the same rule as `indexOf` — from `romsFolder`'s local `gamelist.xml`, then copies its media if newly added or changed. Returns `registry.ErrGameNotFound` if the game has no entry in the system's local `gamelist.xml`. A game with neither a description nor an image is silently skipped, exactly like `ImportFromRomsFolder` (see [`decisions/007`](../decisions/007-skip-empty-games-only-on-import-not-retroactively.md)) — not an error. Unlike `CompleteGame`, a media copy failure is propagated as a real error (mirroring `ImportFromRomsFolder`'s own behavior) rather than counted as a soft per-game failure. `onProgress`, if non-nil, fires once, only if the game was actually imported (added, updated, or unchanged) — see [`decisions/013`](../decisions/013-target-single-game-by-real-disk-path.md)
- `registry.CompleteRomsFolder(reg *Registry, romsFolder, registryFolder string, onProgress func(CompletionEvent)) (processed, completed, failed int, err error)` — the reverse flow: for each system's local `gamelist.xml` in `romsFolder`, fills any field left empty in a local game entry from the matching registry entry (same system + ROM filename, via `indexOf`), never overwriting a field already set locally; for every newly filled media reference, copies the file from `registryFolder` into `romsFolder`. A game with no matching registry entry, or already fully complete, is left untouched (not counted as completed, no event). A media copy error counts that game as failed but does not abort the run; the local `gamelist.xml` is rewritten once per system (via `gamelist.WriteFile`) if any of its games changed. If non-nil, `onProgress` is called only for games that actually had a field filled (whether or not their media copy then succeeded) — untouched games produce no event
- `registry.CompletionEvent` — alias of `ProgressEvent`; one progress notification emitted by `CompleteRomsFolder` and `CompleteGame`
- `registry.CompleteGame(reg *Registry, romsFolder, registryFolder, system, romFilename string, onProgress func(CompletionEvent)) (completed, failed bool, err error)` — the single-game counterpart of `CompleteRomsFolder`: fills the gaps of the one local game entry matching (system, romFilename) — matched by ROM base name via `indexOf`, same as every other registry lookup — from the matching registry entry, then copies any newly filled media reference. Returns `registry.ErrGameNotFound` (reusing the same sentinel as `Remove`) if the game has no entry in the system's local `gamelist.xml`, or no matching entry exists in `reg`; `completed` reports whether a field was actually filled, `failed` whether the subsequent media copy failed (the local `gamelist.xml` is still rewritten with the filled fields in that case). `onProgress`, if non-nil, fires once, only when a field was filled — see [`decisions/013`](../decisions/013-target-single-game-by-real-disk-path.md)
- `registry.Remove(reg *Registry, registryFolder, system, romFilename string) error` — deletes the metadata JSON file and every referenced media file (cover art, video, marquee, thumbnail) for the entry matching (system, romFilename), then removes that entry from `reg`. `romFilename` is matched via `indexOf`, so a full path also works (only its base name matters) but a bare filename (e.g. `Sonic.zip`) suffices even if the ROM originally lived in a subfolder. Returns `registry.ErrGameNotFound`, leaving the registry and disk untouched, if no entry matches. A missing file on disk (already deleted) is treated as success, not an error.
- `registry.ErrGameNotFound` — sentinel error returned by `Remove` when no entry matches the given (system, romFilename)

**Depends on:** [`modules/gamelist.md`](gamelist.md)

**Registry folder layout:** `<registryFolder>/<system>/<romBaseName>.json` holds one game's metadata, alongside `<registryFolder>/<system>/...` media subfolders (e.g. `images/`, `videos/`) mirroring the corresponding ROMs system folder — since media paths in `gamelist.xml` are relative to the system folder and are preserved unchanged when copied. There is no single index file for the whole registry: each game's entry is self-contained in its own file, so a corrupted or malformed entry only affects that one game on load.

**Architecture note:** `ImportFromRomsFolder` is exposed via the `update` CLI command (see [`modules/cli.md`](cli.md)), implemented for backlog item 002. Media copying was added afterward, reversing the initial decision to keep the registry metadata-only, once the need to keep the registry self-contained (independent of the original ROMs folders) was confirmed. The registry was then further split from a single index file into one file per game, for the same self-containment reason. `CompleteRomsFolder` was added for backlog item 003 as the reverse of `ImportFromRomsFolder`: the registry is read-only there, and ROMs folders are the ones written to — see [`decisions/004`](../decisions/004-registry-as-source-to-complete-roms-folders.md). `Remove` was added for backlog item 005, identifying the game to delete the same way every other registry operation does: by (system, ROM filename). Designing `Remove` surfaced an inconsistency between `indexOf` (which compared the full `Game.Path`) and on-disk storage (`gameFileName`, always flat per system, discarding any subfolder prefix) — `indexOf` was changed to match on filename alone, closing a silent-collision gap and letting `remove` take a bare filename instead of a full path — see [`decisions/005`](../decisions/005-match-registry-entries-by-rom-filename-not-full-path.md). `CompleteGame` was added for backlog item 011 to target a single game instead of a whole ROMs folder, reusing `mergeGame`, `copyFilledMedia`, and `indexOf` rather than duplicating `CompleteRomsFolder`'s logic; unlike the batch flow, a missing local entry or missing registry match is reported as `ErrGameNotFound` instead of being silently skipped, since the caller explicitly asked for that one game. `ImportGame` was added symmetrically for backlog item 012 (single-game `update`), reusing `mergeGameEntry`, `copyGameMedia`, and `hasScrapedData` rather than duplicating `ImportFromRomsFolder`'s logic; it keeps `ImportFromRomsFolder`'s existing skip-empty-game business rule as-is (silent skip, not an error) since that rule was already scoped to import as a whole, not just the batch entry point.

A second `/vibe:review` pass (2026-07-02) fixed a real bug found by `review-tests`: `gameFileName` stripped the ROM's file extension when naming a game's JSON file, but `indexOf` still matched on the full filename including extension, so two ROMs differing only by extension (e.g. `Sonic.zip` and `Sonic.iso`) collided on the same on-disk file with no warning — see [`decisions/014`](../decisions/014-dedupe-by-extension-stripped-filename-too.md). Both now derive their key from a shared `gameID(path string) string` helper. That same pass also: renamed the unexported `importGame` method to `mergeGameEntry` and the unexported `completeOneGame` helper to `fillGameFromRegistry`, so they no longer differ from the exported `ImportGame`/`CompleteGame` functions by capitalization alone; extracted `importSystemGames` and `completeSystemGames` from `ImportFromRomsFolder`/`CompleteRomsFolder`'s inner per-system loops (mirroring the earlier `importOneGame`/`fillGameFromRegistry`/`findGameByFilename`/`loadSystemEntries` extractions from the first pass); unified `CompletionEvent` as a type alias of `ProgressEvent` (the two were structurally identical); and introduced a `mediaFields` table (accessors for Image/Video/Marquee/Thumbnail) so `Remove`, `copyGameMedia`, and `copyFilledMedia` iterate one shared list instead of each hardcoding the same four fields separately.
