## Development workflow (Vibe Coding)

The user is **Product Owner only**. They describe requirements and evaluate results — they do not write or manually test the code.

### Definition of Done

A task is done ONLY if ALL of the following are true:

- [ ] The implementation code is written
- [ ] Tests covering the happy path exist and pass
- [ ] Tests covering edge cases and error paths exist and pass
- [ ] `gofmt -l .` returns no files (clean formatting)
- [ ] `go vet ./...` reports no warnings
- [ ] `go test ./...` passes entirely
- [ ] No debug artifacts left in the code (debug `fmt.Println`, etc.)

**Never present a result to the user if tests are failing.**

### TDD workflow

For every feature or fix:

1. **Write tests first** — describe the expected behavior via tests before writing the implementation
2. **Confirm the tests fail** — run `go test ./...` and verify the new tests fail (red)
3. **Write the implementation** — minimal code to make the tests pass
4. **Confirm the tests pass** — re-run `go test ./...` and verify everything passes (green)
5. **Refactor if needed** — clean up while keeping tests green
6. **Run the linter** — `gofmt -l .` and `go vet ./...`, fix any deviation
7. **Present the result** — summarize what was done, what was tested, and the test output

### Bug fix workflow

1. **Reproduce first in a test** — write a failing test that captures the bug
2. Fix the bug
3. Confirm the test passes
4. Present the result along with the name of the test now covering the bug

### Self-correction loop

If tests or lint fail:
- Do NOT ask the user for help
- Diagnose the failure, fix it, re-run
- Repeat until green (3 self-correction attempts maximum)
- Only escalate to the user if the failure reveals an ambiguous requirement

## Test conventions

- Test location: `*_test.go` files colocated with the code (e.g. `internal/cli/cli_test.go`)
- Test framework: `testing` (Go standard library)
- One test file per source file
- Test names must describe a behavior: `TestExecute_UnknownCommand_ReturnsErrorCode` rather than `TestExecute2`
- Always cover: happy path + at least 2 edge cases + invalid input/error

## Constraints

- Never dead code or unused imports
- Never hardcoded secrets — use environment variables
- Never skip a test to meet a deadline — fix the code instead
- Style is enforced by tooling, not convention — always run `gofmt -l .` and `go vet ./...` before presenting a result
